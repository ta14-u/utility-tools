# QRコードの漢字モード

## 概要
QRコードには日本語の文字を効率よく格納するための「漢字モード」が用意されています。
Shift_JIS の2バイト文字を13ビットに圧縮して記録できるため、8ビットバイトモードよりも1文字あたり約3ビット分効率が良くなります。

## 文字範囲と13ビット化の考え方
漢字モードで扱うShift_JISの2バイト文字は、次の範囲にあるものが対象です。

- `0x8140`〜`0x9FFC`
- `0xE040`〜`0xEBBF`

符号化の手順は次のとおりです。

1. Shift_JISコードが `0x8140`〜`0x9FFC` の場合、`0x8140` を引く
2. Shift_JISコードが `0xE040`〜`0xEBBF` の場合、`0xC140` を引く
3. 得られた値の上位バイトに `0xC0` を掛け、下位バイトを足す
4. `value = (上位バイト * 0xC0) + 下位バイト` を計算

この `value` は最大で `0x1FFF` になり、13ビットで表現できます。

## 例: 「技術書典」の符号化
「技術書典」を漢字モードで符号化する例です。各文字をShift_JISで表したあと、上記の計算で13ビットに変換します。

```
1000              （漢字モード）
00000100          （文字数=4）
0011110011010     （「技」= 8B5A → (8B-81)*C0 + (5A-40) = 079A）
0101010110000     （「術」= 8F70 → (8F-81)*C0 + (70-40) = 0AB0）
0101011010001     （「書」= 8F91 → (8F-81)*C0 + (91-40) = 0AD1）
0110110010100     （「典」= 9354 → (93-81)*C0 + (54-40) = 0D94）
```

## 8ビットバイトモードとの比較
同じ文字列を8ビットバイトモードで格納する場合、Shift_JISの2バイトをそのまま格納するため、1文字あたり16ビットが必要になります。
漢字モードでは13ビットで済むため、長い日本語文字列ほど差が大きくなります。

## 英数字が混在する場合の扱い
入力が英数字のみで構成され、かつ英字が大文字で記号が「`$ % * + - . / :`」とスペースの範囲に収まる場合は、英数字モードが最も効率的です。
一方、英小文字や上記以外の記号、Shift_JISの2バイト漢字が混在する場合は、1つのモードだけでは収まらないため、複数のモードを切り替えて符号化することができます。
モードを切り替える際は、切り替え地点ごとにモード指示子を挿入します。混在文字列の最適な分割方法は仕様で定義されていますが、本ドキュメントでは詳細には踏み込みません。

## 注意点
- 漢字モードで扱えるのは上記範囲のShift_JISの2バイト文字のみです。
- 1バイト文字は漢字モードでは格納できません。
- 最大で1,817文字を格納できます（型番40・誤り訂正レベルLの場合）。
- 8ビットバイトモードでShift_JISを使う場合は、ECIモードで文字コードを明示しておくと誤解釈を避けられます。

## QRコードのセグメント化（混在モード）の詳細仕様

### セグメント化の概要
QRコードは複数のエンコーディングモードを混在させることができます。これを**セグメント化（Segmented Encoding）**または**混在モード（Mixed Mode）**と呼びます。

テキストを複数のセグメントに分割し、各セグメントが異なるモードを使用することで、データをより効率的にエンコードできます。

### セグメントの構造
各セグメントは以下の3つのフィールドで構成されます：

1. **モードインジケータ** (4ビット)
   - Numericモード: `0001`
   - Alphanumericモード: `0010`
   - Byteモード: `0100`
   - Kanjiモード: `1000`

2. **文字数インジケータ** (可変長)
   - モードとQRコードのバージョンによって長さが変わります
   - 例: Kanjiモードでバージョン1-9の場合は8ビット

3. **データ** (可変長)
   - 実際のエンコードされたデータ

### セグメント化の例
例えば `"123456ABC日本語"` のような混在テキストは以下のようにセグメント化できます：

```
セグメント1: Numericモード
  モードインジケータ: 0001
  文字数: 6
  データ: "123456" をNumericモードでエンコード

セグメント2: Alphanumericモード
  モードインジケータ: 0010
  文字数: 3
  データ: "ABC" をAlphanumericモードでエンコード

セグメント3: Kanjiモード
  モードインジケータ: 1000
  文字数: 3
  データ: "日本語" をKanjiモードでエンコード
```

全体をByteモードで扱うよりも、各部分に最適なモードを使用することでビット数を削減できます。

### モード切り替えのコストと最適化
ただし、モードを切り替えるたびにモードインジケータ(4ビット)と文字数インジケータ(可変長)が必要になるため、頻繁な切り替えは逆効果になる場合があります。

最適なセグメント分割を見つけるには、**動的計画法**を用いたアルゴリズムが使われます。これにより、総ビット数が最小になるような分割を計算できます。

### ライブラリの対応状況

#### Java版ZXing
Java版のZXingライブラリでは、`EncodeHintType.QR_CODE_COMPACT` を `true` に設定することで、`MinimalEncoder` が有効になり、自動的に最適なセグメント分割が行われます。

```java
Map<EncodeHintType, Object> hints = new HashMap<>();
hints.put(EncodeHintType.QR_CODE_COMPACT, true);
hints.put(EncodeHintType.ERROR_CORRECTION, ErrorCorrectionLevel.L);
hints.put(EncodeHintType.CHARACTER_SET, "UTF-8");
QRCode qrCode = Encoder.encode(content, ErrorCorrectionLevel.L, hints);
```

#### zxing-js (TypeScript版)
zxing-jsライブラリでは、現時点で以下の制限があります：

- `Encoder.chooseMode()` メソッドはコンテンツ全体に対して**単一のモード**を選択する設計
- `QR_CODE_COMPACT` ヒントや `MinimalEncoder` に相当する機能は未実装
- セグメント化による複数モード混在は**未対応**

そのため、`aaaaa,\nあああああ` のような混在テキストをエンコードする場合：
- ASCII文字 (`a`, `,`, `\n`) はKanjiモードで表現できない
- 全体として `chooseMode()` が Byteモードを選択
- `CHARACTER_SET: "SJIS"` ヒントによりShift_JISでエンコードされるが、モードはByteのまま
- 結果として「Shift_JISでエンコードされたByteモードのQRコード」が生成される

#### 現在の実装での対処方法
本プロジェクトでは、zxing-jsの制限を考慮して、**Kanjiモードで生成する際はShift_JIS 2バイト文字のみを受け入れる**バリデーションを実装しています。

混在テキストをエンコードする場合は、Kanjiモードのチェックを外して通常モード（自動選択）で生成してください。
